## Контекст

Decartes — небольшая TypeScript-библиотека для canvas drawing engine с минимальным README, компактным экспортируемым API в `src/decartes.ts` и командами разработки, определенными в `package.json`. В репозитории также есть OpenSpec workflow assets, но нет проектного руководства, которое объясняет coding agents, как ориентироваться перед изменением исходного кода, документации, тестов или планировочных артефактов.

Изменение относится только к документации. Оно должно улучшить ориентацию в репозитории, не меняя runtime-поведение, результат сборки, package metadata или публичные экспорты.

## Цели / Не цели

**Цели:**

- Дать coding agents и контрибьюторам понятный документ, с которого стоит начинать работу.
- Организовать текущие факты о проекте: назначение, структуру исходников, точки входа публичного API, команды сборки и тестов, OpenSpec workflow.
- Добавить ссылку на новую документацию из существующего README, чтобы она была обнаруживаема не только через agent-specific tooling.
- Сохранить документацию достаточно краткой, чтобы ее было легко поддерживать по мере развития небольшой библиотеки.

**Не цели:**

- Менять runtime-поведение Decartes, публичный API, семантику рендеринга или package exports.
- Заменять документацию процесса OpenSpec или файлы skills.
- Добавлять генерацию API reference или новые зависимости для документации.
- Превращать README в полноценный мануал.

## Решения

- Добавить корневой `AGENTS.md` как основную точку входа для агентов.
  - Обоснование: agent guidance на уровне корня легко находят локальные инструменты и люди перед внесением изменений.
  - Рассмотренная альтернатива: поместить все руководство в `docs/`. Это делает корень чище, но ухудшает обнаруживаемость стартовых инструкций для агентов.

- Добавить вспомогательную документацию в `docs/` для стабильных деталей ориентации по проекту.
  - Обоснование: долгоживущие объяснения, например архитектура, публичные экспорты и команды разработки, проще поддерживать в отдельных сфокусированных файлах, чем в одном большом корневом документе.
  - Рассмотренная альтернатива: держать весь контент в `AGENTS.md`. Это уменьшает количество файлов, но делает будущие обновления шумнее и хуже сканируемыми.

- Ограничить изменения README короткой ссылкой на документацию.
  - Обоснование: сейчас README работает как минимальное введение в пакет, поэтому изменение должно сохранить эту роль и при этом сделать новые документы обнаруживаемыми.
  - Рассмотренная альтернатива: переписать README как основной contributor guide. Это смешало бы аудитории package-facing и agent-facing документации.

- Не добавлять новые зависимости или сгенерированную документацию.
  - Обоснование: в репозитории сейчас нет documentation tooling, а нужный контент можно поддерживать как обычный Markdown.
  - Рассмотренная альтернатива: генерировать API-документацию из TypeScript declarations. Это добавляет настройку и стоимость поддержки, не решая текущий пробел в ориентации.

## Риски / Компромиссы

- Документация может устареть при изменении экспортов, скриптов или структуры -> Добавить явное правило поддержки: обновлять документацию вместе с такими изменениями.
- Дополнительные файлы могут казаться избыточными для небольшой библиотеки -> Держать корневое руководство коротким и выносить только стабильный reference-контент в отдельные сфокусированные документы.
- Agent-specific формулировки могут путать обычных контрибьюторов -> Использовать простой contributor-friendly язык и ограничить tool-specific инструкции ссылками на workflow.

## План миграции

1. Добавить `AGENTS.md` со стартовыми инструкциями для изменений.
2. Добавить сфокусированные документы в `docs/` для структуры репозитория, ориентации по публичному API, команд и OpenSpec workflow.
3. Обновить `README.md`, добавив ссылки на документацию для агентов и контрибьюторов.
4. Проверить ссылки в документации и подтвердить, что исходный код и поведение пакета не изменились.

Rollback простой: удалить добавленные документационные файлы и ссылки из README.
